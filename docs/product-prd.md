# WebRelay 产品 PRD

## 产品定位

WebRelay 是一个面向常用网站的 Chrome 网页信息抽取插件。它的核心价值不是每次临时让模型抽取一次内容，而是帮助用户把某个网站上的信息抽取需求沉淀成可复用配置，之后可以稳定地、本地地、一键运行。

Codex SDK 是配置创建与修复阶段的智能能力来源。它负责分析页面、生成抽取规则、生成本地输出格式、根据用户反馈迭代，以及在配置失效时辅助修复。日常使用路径应尽量不依赖 Codex，也不依赖后端。

## 目标用户

- 经常从同一类网站重复收集信息的用户。
- 需要把网页内容整理成固定格式的用户，例如 Markdown、JSON、CSV、自定义文本。
- 能描述自己想要什么信息，但不想手写 CSS selector 或抽取逻辑的用户。
- 希望结果可以直接复制或下载，而不是每次手动整理的用户。

## 核心问题

- 网页 DOM 结构不稳定，人工复制粘贴重复且容易出错。
- 一次性 LLM 抽取不适合长期重复使用。
- 手写 selector 门槛高，且网站更新后容易失效。
- 用户既需要智能创建配置，也需要低摩擦的日常执行入口。
- 结果获取动作应该可以预设，而不是每次运行后再手动决定。

## 产品目标

- 让用户能基于当前网页创建一套可复用抽取配置。
- 让用户在保存前看到真实预览结果和 debug 信息。
- 让用户通过 Codex Studio 交互式完善抽取内容和输出格式。
- 让失效配置可以带着旧配置、失败信息和当前页面重新修复。
- 让保存后的配置可以在 Daily Run 中本地执行，并自动复制、下载或同时执行两者。
- 为快捷键无弹窗执行预留清晰行为：运行当前站点上次成功使用的配置。

## 非目标

- 近期不做云同步、多人共享、托管后端。
- 近期不做外部 API 投递、数据库写入、webhook、第三方应用自动化。
- 不允许 Codex 生成任意注入网页执行的脚本。
- 不持久化保存抽取结果内容。
- 不把后端迁移到 OpenAI API；后端继续使用 `@openai/codex-sdk` 和本地 Codex CLI 登录。

## 核心概念

- Site：站点分组，代表一个网站或一组 URL 匹配规则。
- Configuration：某个站点下的一套可复用配置。
- Recipe：封闭 JSON 抽取规则，由扩展运行时执行。
- Transform：受限本地格式化逻辑，把抽取数据转成字符串输出。
- Action preset：配置保存的默认结果动作，支持复制、下载、复制加下载。
- Daily Run：无 Codex 参与的日常执行界面。
- Codex Studio：有 Codex 参与的生成、预览、反馈、修复、保存工作台。

## 用户故事

### 首次创建配置

作为用户，我在目标网站上点击 Generate，即使没有写清楚需求，也希望 WebRelay 能先分析当前页面，提出可抽取的信息，运行预览，并让我保存成可复用配置。

验收标准：

- 扩展能获取当前页面 DOM snapshot。
- 后端通过 Codex SDK 生成结构化抽取 artifact。
- 扩展在当前页面本地运行 recipe。
- 用户保存前能看到预览数据和 debug 信息。
- 保存后配置归入当前站点。

### 带需求创建配置

作为已经知道自己想要什么的用户，我希望先输入抽取需求和输出格式，再让 Codex 生成配置。

验收标准：

- 用户可以在生成前输入抽取 intent。
- Codex 使用 intent 和 DOM snapshot 生成 recipe。
- 如果用户提出输出格式要求，Codex 可以生成受限 transform。
- 用户可以继续反馈、预览、保存。

### 修复失效配置

作为用户，当某个已保存配置运行失败时，我希望直接进入同一个 Codex Studio，通过旧配置和失败信息帮助 Codex 修复，而不是重新从零创建。

验收标准：

- 修复请求包含旧 recipe、原始 intent、当前 DOM snapshot、执行 debug、失败原因、已有输出设置。
- 修复后的 recipe 必须先预览。
- 用户决定是否覆盖旧配置或另存为新配置。

### 日常执行

作为回访用户，我希望打开插件后直接选择当前网站的某套配置，运行并得到结果，不需要 Codex 参与。

验收标准：

- 插件根据当前 URL 展示匹配配置。
- 运行配置时在 content script 中本地执行 recipe。
- 若配置有 transform，则生成格式化输出。
- 根据 action preset 执行复制、下载或两者。

### 快捷键执行

作为高频用户，我希望按快捷键即可运行当前站点上次成功使用的配置，并自动完成预设动作。

验收标准：

- 快捷键能根据当前 URL 找到匹配站点。
- 优先选择该站点上次成功使用的配置。
- 本地运行 recipe 和 transform。
- 自动执行该配置保存的 action preset。
- 不保存抽取结果内容。

## 成功标准

- 用户能清楚区分“Codex 辅助创建/修复”和“本地日常执行”。
- 一套配置能说明它抽取什么、如何输出、默认执行什么动作。
- 失效配置可以借助 debug 上下文修复，而不是只能重建。
- 已保存配置在后端未运行时仍可完成日常执行。
- 文档能指导后续重构，不需要重新讨论核心产品边界。

## 阶段规划

### MVP

- 统一术语和产品文档。
- 保持现有 recipe 与 transform 安全边界。
- 明确 Daily Run 与 Codex Studio 的界面职责。
- 明确配置中保存 action preset。
- 保存前以预览加用户确认作为质量门槛。

### V1

- 实现站点分组和每站点多套配置。
- 将 Daily Run 作为 popup 默认日常入口。
- 将生成、反馈、修复迁移到 Codex Studio。
- 支持复制、下载、复制加下载的默认动作。

### V2

- 支持快捷键运行当前站点上次成功配置。
- 增加配置管理能力：重命名、复制、删除、设为默认。
- 增加预览质量提醒，但最终仍由用户确认保存。
